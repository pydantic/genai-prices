# Code Spec: Phase 8 Usage Inference

**This implements the prose spec in [spec](spec.md), which is the primary source of truth.**
**Baseline:** this phase starts after Phase 7, with both runtimes using explicit-only missing-usage checks and active-snapshot guardrails.

**Phase 8 replaces conservative missing-usage failures with demand-driven inference.** _(implements "Phase 8 adds missing-usage inference after the registry architecture is complete")_
The implementation should remove the Phase 1/2 rule that missing priced ancestors and overlaps always fail when descendant or overlapping usage suggests they may be needed. Instead, the runtime attempts to infer the missing value. It still raises if the value is contradictory or underdetermined.

**Usage containers still store reported values only.** _(implements "Inference is demand-driven, not construction-time normalization", "Reported values remain authoritative")_
Python `Usage` and JavaScript normalized usage keep their reported-value storage. Do not add eager normalization, inferred-value caching, or explicit-versus-inferred fields to public usage objects. Addition, equality, representation, and serialization continue to operate on reported values.

**Missing reads call an inference helper.** _(implements "Missing values are inferred only when uniquely determined")_
When a registered usage key is missing, both runtimes call a helper equivalent to:

```text
inferUsageValue(requestedUnit, reportedValues, registry) -> number
```

The helper receives the requested `UnitDef`, the reported usage values in that family, and registry relationship indexes. If no relevant descendant data exists, it returns zero. If the reported descendants uniquely determine the requested unit's total, it returns that total. If not, it raises a user-facing error.

**The conceptual inference model is atom-based.** _(implements "Missing values are inferred only when uniquely determined", "Inference is registry-derived for all unit families")_
For a missing requested unit `U`:

1. Collect stored strict descendants of `U` in the same family. Ignore reported units outside `U`'s subtree.
2. Build atomic regions generated by those descendants. For each dimension key below `U`, an atom records either one concrete reported value for that key or "other". Keep only atoms covered by at least one reported descendant; atoms outside all reports are zero for this missing-read inference.
3. Create one non-negative variable per remaining atom.
4. For each reported descendant `D = value`, add an equation: the sum of atom variables contained in `D` equals `value`.
5. If the equations have no non-negative solution, raise a contradiction error.
6. Let `total(U)` be the sum of all remaining atom variables. If every non-negative solution gives the same `total(U)`, return that value.
7. Otherwise raise an underdetermined-value error.

This is a behavioral model, not a mandate to add a heavyweight solver dependency. The built-in and custom registries are expected to be small. Implementations may use exact rational arithmetic, exhaustive basis enumeration, full inclusion-exclusion plus conservative fallback, or another registry-derived method if tests prove the same observable behavior.

**Pricing reads missing values through the inference helper.** _(implements "Pricing may use inference where earlier phases raised")_
Dimension-driven decomposition should call the usage-value read path for priced units. Missing priced ancestors, missing priced overlaps, and missing tier thresholds are inferred when possible instead of failing solely because they were omitted. Negative exclusive leaf values still raise as impossible usage data.

**Tiered pricing infers only the threshold it needs.** _(implements "Tier thresholds use the same inference rule")_
If no selected price uses `TieredPrices`, pricing continues to pass a neutral threshold because non-tiered prices ignore it. If a selected tiered price contributes non-zero usage and explicit `input_tokens` is absent, pricing reads `input_tokens` through the inference helper. It raises if the total cannot be inferred coherently.

**Tests cover unique, contradictory, and underdetermined inference.** _(implements "Missing values are inferred only when uniquely determined", "Pricing may use inference where earlier phases raised")_
Add cross-language tests for: one descendant determining an ancestor; overlapping descendants plus reported join determining an ancestor; disjoint descendants summing to an ancestor; overlapping descendants without a reported/determined join raising as underdetermined; descendant counts that cannot fit inside each other raising as contradictory; direct reads returning stored values without auditing descendants; inferred values not appearing in equality, representation, addition, or serialization; tier thresholds inferred only when coherent; runtime custom units participating in the same inference after activation.
