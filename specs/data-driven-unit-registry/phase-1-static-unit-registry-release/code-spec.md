# Code Spec: Phase 1 Static Unit Registry Release

**This implements the prose spec in [spec](spec.md), which is the primary source of truth.**

**Phase 1 consolidates the implemented Python, JavaScript, shared-registry, and polish work into one release.** _(implements "Phase 1 must be independently shippable and releasable", "Pricing accuracy is the Phase 1 product outcome")_
The implementation keeps the current registry-backed pricing, usage, validation, generated authoring schema, direct Python `ModelPrice` storage, raw-provider normalization, and registry-driven CLI behavior. It changes the current branch's remote-payload and registry-lifecycle implementation to the static v2 contract described here.

**Source units remain flat usage-keyed YAML.** _(implements "Units are repo-defined data used by handwritten runtime code")_
Maintain `prices/units.yml` as the source registry. Each unit carries `per`, optional `price_key`, and `dimensions` with `family`. The complete built-in token lattice includes the intermediate and join units required for input/output, cache read/write, and supported modality overlap, including image-token units. `requests` remains the explicit pricing-only unit.

**Build/export validation remains reusable and authoritative.** _(implements "Build/export validation is the authoritative publication boundary")_
Keep a build helper that accepts parsed providers plus raw unit data and returns a validated `UnitRegistry`. It validates unit structure, public key safety, provider price keys, ancestor and join coverage, extractor destinations, and the documented built-in naming conventions before publishing artifacts. Runtime construction of generated unit data indexes trusted content rather than repeating publication validation.

**Python and JavaScript each construct one generated registry.** _(implements "The Phase 1 registry is static for the lifetime of the installed package", "`UnitRegistry` is immutable indexed metadata after construction")_
Python's private registry accessor lazily constructs and returns the bundled registry from `genai_prices.data_units.unit_data`. JavaScript constructs one module-level registry from `dataUnits.ts`. Remove production active-registry setters, candidate-registry installation, restoration, reset-on-stop behavior, and comments or tests that reserve those paths for cache invalidation. Validation helpers may continue accepting explicit registry arguments for build code and isolated tests.

`UnitRegistry` construction precomputes usage-key, price-key, dimension, ancestor, join, all-key, reported-key, and reported-key-order indexes. Python does not recreate `frozenset` or ordered tuple collections for every `Usage` operation. JavaScript internal validation may read stored relationship sets directly when it does not need a defensive public copy.

**Generated provider and unit modules remain separate.** _(implements "Generated package providers and units have separate inputs")_
`prices/src/prices/package_data.py` generates:

- `packages/python/genai_prices/data.py` from the v2 provider array;
- `packages/python/genai_prices/data_units.py` from validated `prices/units.yml` data;
- `packages/js/src/data.ts` from the v2 provider array;
- `packages/js/src/dataUnits.ts` from the same validated unit data.

Generated provider files do not embed registries, and generated unit files do not import provider data. Generated files contain data only.

**V1 remote artifacts are restored rather than regenerated by this phase.** _(implements "The existing v1 JSON artifacts do not change in Phase 1")_
Restore `prices/data.json`, `prices/data_slim.json`, `prices/data.schema.json`, and `prices/data_slim.schema.json` to the branch-point contents. Phase 1 build/package-data commands must not rewrite those files as a side effect. Existing URLs and old package behavior remain unchanged.

**The build emits a separate v2 provider array and schema.** _(implements "Phase 1 publishes provider-array `data_v2.json`")_
Write current publishable providers to `prices/data_v2.json` with the same top-level array shape as v1. Generate `prices/data_v2.schema.json` from the current provider model and registry-derived allowed price/extractor keys. Do not include `units`, validation state, or cached plans in either artifact. A slim v2 artifact is not required in Phase 1.

**Python auto-update parses only v2 provider arrays.** _(implements "Phase 1 auto-updaters use only `data_v2.json`")_
Change the Python default update URL to `prices/data_v2.json`. `UpdatePrices.fetch()` parses the response as providers using the internal raw-provider normalization path and returns a provider-only `DataSnapshot`. It does not construct or install a registry. `UpdatePrices.stop()` clears the custom provider snapshot without touching unit state.

**JavaScript auto-update parses only v2 provider arrays.** _(implements "Phase 1 auto-updaters use only `data_v2.json`")_
Change the JavaScript remote URL to `prices/data_v2.json`. `setProviderData` accepts `null`, a provider array, or a promise resolving to either. It validates runtime provider extractor destinations against the bundled registry before activation, keeps existing provider data on failure, and never accepts or activates a wrapped unit payload in Phase 1.

**Python `Usage` and `ModelPrice` retain the completed direct-storage design.** _(implements "Python `Usage` stores reported registry values directly", "Python `ModelPrice` uses direct public attribute storage")_
`Usage` stores reported values in `__dict__`, filters them through bundled reported-key indexes for equality/representation/addition, and performs explicit-only missing reads through bundled relationships. `ModelPrice` stores candidate prices as direct public attributes, exposes absent bundled price keys as `None`, excludes subclass-only custom state from standard pricing, and preserves the original usage object for overrides. Runtime provider normalization converts raw price mappings into `ModelPrice` before validating surrounding dataclasses.

**JavaScript retains open records and direct registry-aware usage reads.** _(implements "JavaScript retains plain usage and model-price objects")_
`Usage` and `ModelPrice` remain record types. `normalizeUsage(...)` is used for extractor and normalized-return paths, while standard pricing reads the caller object directly. Missing-value safety, tier thresholds, request pricing, decomposition, and aggregation match Python behavior.

**Both runtimes validate and price through one per-call structural collection.** _(implements "Phase 1 removes repeated structural work from the pricing hot path", "JavaScript model validation materializes key and unit collections once")_
For standard pricing, collect the selected model's effective non-null price entries once and resolve their registered units once. Unknown base price keys still raise, while subclass-only custom state remains excluded in Python. Use the resolved collection for:

- price-key, ancestor, and join validation;
- selecting non-`requests` decomposition units;
- determining whether any current price value is tiered;
- reading current price values;
- normalization and result aggregation.

Do not scan all registry units separately for key collection, unit collection, and tier detection. Independently public/internal validation helpers may preserve their current signatures, but the standard pricing composition shares intermediate collections rather than invoking redundant full helper pipelines.

**Decomposition stays direct and uncached.** _(implements "Phase 1 deliberately excludes runtime-updated units and performance caches")_
Compute exclusive values from the resolved priced-unit collection and current explicit usage on every calculation. Do not add model-object caches, registry validation IDs, weak references, fingerprint state, prepared pricing plans, or decomposition coefficient caches.

**Authoring schemas and CLI remain registry-driven.** _(implements "Provider YAML authoring derives allowed keys from the registry", "CLI price presentation is registry-driven")_
Provider YAML schema generation derives price-key and extractor-destination suggestions from the validated source registry. CLI price discovery iterates stored/effective model prices and derives labels plus normalization from unit metadata. Existing familiar labels may remain compatibility aliases, but ordinary new units do not add hardcoded branches.

**Benchmarks are reproducible but non-gating.** _(implements "Performance evidence covers representative price shapes")_
Add or retain a lightweight benchmark entry point for Python and JavaScript direct/public pricing with empty, one/two-key, four-key overlap, and five/six-key complex price shapes. Document enough invocation context to compare the pre-cleanup branch with the single-collection result. Do not make wall-clock thresholds part of normal tests.

**Tests replace wrapped-payload lifecycle coverage with static-v2 coverage.** _(implements "Tests prove the complete releasable contract")_
Keep registry, validation, usage, decomposition, dynamic-price, modality-price, schema, CLI, custom-pricing, and cross-runtime parity coverage. Replace tests for wrapped roots, candidate registry installation, registry rollback, and reset-on-stop with tests that:

- v1 artifacts are unchanged;
- v2 and its schema are provider-array artifacts with new registered keys;
- package unit data comes from the source registry rather than v2;
- successful and failed provider updates preserve the bundled registry identity;
- both auto-updaters point at v2;
- the hot path reuses one structural collection without behavior drift.
