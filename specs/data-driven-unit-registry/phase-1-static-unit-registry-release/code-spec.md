# Code Spec: Phase 1 Static Unit Registry Release

**This implements the prose spec in [spec](spec.md), which is the primary source of truth.**

**Phase 1 consolidates the implemented Python, JavaScript, shared-registry, and polish work without changing supported entry points.** _(implements "Phase 1 preserves the shared registry and pricing invariants", "Phase 1 must be independently shippable and releasable", "Pricing accuracy is the Phase 1 product outcome", "Phase 1 preserves supported consumer behavior unless accurate registry pricing makes that impossible", "Public pricing and snapshot APIs keep their callable shape", "Manual Python pricing extensions remain supported")_
Context: keep the branch's registry-backed pricing, usage, validation, generated authoring schema, direct Python `ModelPrice` storage, raw-provider normalization, and registry-driven CLI behavior. Change only the remote-payload and registry-lifecycle implementation needed for the static v2 contract. The public functions and methods named in the prose spec retain their argument and return shapes; internal helpers may gain explicit registry or resolved-price parameters.

**Source units remain flat usage-keyed YAML and define the complete pricing graph.** _(implements "Units are repo-defined data used by handwritten runtime code", "Dimensions define containment, overlap, and cost aggregation", "Every priced usage value lands in exactly one bucket", "`requests_kcount` remains an explicit one-request pricing exception", "`web_searches_kcount` prices reported web searches")_
Maintain `prices/units.yml` as the source registry. Each unit carries `per`, optional `price_key`, and `dimensions` with `family`. The complete built-in token lattice uses mutually exclusive `token_type` values for cache read/write, input tool-use, output reasoning, and output citation categories. It includes the intermediate and join units required for input/output and supported modality overlap, without cross-category joins. `requests` remains the pricing-only unit with `requests_kcount` and `per: 1_000`; `web_searches` is the reportable singleton `tool_calls` family unit with `web_searches_kcount` and `per: 1_000`. `packages/python/genai_prices/decompose.py` and `packages/js/src/decompose.ts` continue computing exclusive buckets from `UnitDef.dimensions`; they receive only the priced non-request units resolved for the current calculation.

**Named build/export validation functions remain reusable and authoritative.** _(implements "Price data is complete while usage data may be incomplete", "Repo-authored prices avoid redundant equal-rate descendants", "Build/export validation is the authoritative publication boundary", "Model prices are still validated on use", "Extractor destinations are externally reported bundled-registry usage keys", "Provider YAML authoring derives allowed keys from the registry")_
`prices/src/prices/export_validation.py` owns `validate_units(raw_units: Mapping[str, Mapping[str, object]]) -> UnitRegistry` and `validate_export_payload(providers: list[Provider], raw_units: Mapping[str, Mapping[str, object]]) -> UnitRegistry`. `validate_units(...)` enforces unique usage keys, price keys, and dimension sets; the shared public-identifier, keyword, leading-underscore, and prototype-hazard restrictions; one `per` value per family; exact interval closure; and join-closedness. `prices/src/prices/package_data.py` owns `validate_provider_model_prices(providers: list[Provider], registry: UnitRegistry) -> None` and `validate_provider_extractor_destinations(providers: list[Provider], registry: UnitRegistry) -> None`. The build calls these functions as part of provider-data publication; repo-data regression tests enforce the equal-rate-child convention and exact built-in naming patterns. Runtime registry construction indexes trusted generated units without repeating publication validation, while `packages/python/genai_prices/validation.py::validate_model_price(price_keys: set[str], registry: UnitRegistry) -> None` and `packages/js/src/validation.ts::validateModelPrice(priceKeys: Iterable<string>, registry: UnitRegistry): void` still validate a selected custom/runtime model price before decomposition.

**Python and JavaScript each construct one generated registry.** _(implements "The Phase 1 registry is static for the lifetime of the installed package", "`UnitRegistry` is immutable indexed metadata after construction")_
`packages/python/genai_prices/units.py` retains `UnitDef`, `UnitRegistry(raw_units: Mapping[str, Mapping[str, object]])`, and `_get_registry() -> UnitRegistry`; `_get_registry()` lazily constructs `_bundled_registry` from `genai_prices.data_units.unit_data` and always returns that object. Remove `_active_registry` and `_set_registry`. `packages/js/src/units.ts` constructs one `generatedRegistry` with `new UnitRegistry(rawUnits: RawUnitsDict)` from `dataUnits.ts`; its getters always read that object, and `setActiveRegistry` plus mutable `activeRegistry` state are removed. Validation helpers continue accepting an explicit `UnitRegistry` for build code and isolated tests.

`DataSnapshot` remains provider-only. `set_custom_snapshot(snapshot: DataSnapshot | None) -> None` replaces or clears only provider data; snapshots do not carry or replace the bundled registry, and activation does not bulk-validate model prices. Standard base pricing continues validating only the selected model price on use.

`UnitRegistry.__init__` and the TypeScript constructor precompute usage-key, price-key, dimension, ancestor, join, all-key, reported-key, and reported-key-order indexes. Python exposes `units` as a read-only mapping and stores protected `_reported_usage_keys: frozenset[str]` and `_reported_usage_keys_in_order: tuple[str, ...]` indexes once instead of recreating them for every `Usage` operation. JavaScript keeps its map/set indexes in ECMAScript-private fields, exposes only lookup, read-only iteration, membership, and defensive-copy methods needed by runtime consumers, and freezes each constructed `UnitDef` and its dimensions so callers cannot mutate registry metadata.

**Generated provider and unit modules remain separate.** _(implements "Generated package providers and units have separate inputs")_
`prices/src/prices/package_data.py` generates:

- `packages/python/genai_prices/data.py` from the v2 provider array;
- `packages/python/genai_prices/data_units.py` from validated `prices/units.yml` data;
- `packages/js/src/data.ts` from the v2 provider array;
- `packages/js/src/dataUnits.ts` from the same validated unit data.

Generated provider files do not embed registries, and generated unit files do not import provider data. Generated files contain data only.

**V1 remote artifacts are no longer build outputs.** _(implements "The existing v1 JSON artifacts remain outside Phase 1 build output")_
`prices/src/prices/build.py` and `prices/src/prices/package_data.py` must not open `prices/data.json`, `prices/data_slim.json`, `prices/data.schema.json`, or `prices/data_slim.schema.json` for writing. Existing URLs, provider-array contracts, and old package behavior remain unchanged.

**The build emits a separate v2 provider array with a frozen unit vocabulary.** _(implements "Phase 1 publishes provider-array `data_v2.json`", "The v2 unit vocabulary is frozen")_
`prices/src/prices/build.py` writes current publishable providers to `prices/data_v2.json` with the same top-level array shape as v1 and writes `prices/data_v2.schema.json` from the provider model plus the Phase 1 registry's allowed price and extractor keys. `prices/src/prices/package_data.py::package_data()` reads providers from `data_v2.json` and units independently from `prices/units.yml`. Neither v2 artifact contains `units`, validation state, or cached plans, and no slim v2 artifact is generated. Before Phase 2 exists, price releases may update v2 only when every price key and extractor destination is accepted by its checked-in schema; adding a unit requires v3 instead.

**Python auto-update parses only v2 provider arrays.** _(implements "Phase 1 auto-updaters use only `data_v2.json`")_
In `packages/python/genai_prices/update_prices.py`, set `DEFAULT_UPDATE_URL` to the raw `prices/data_v2.json` URL. `UpdatePrices.fetch()` requires a JSON array, passes it to `packages/python/genai_prices/types.py::_providers_from_raw`, validates extractor destinations against `_get_registry()`, and returns a provider-only `DataSnapshot`. It never imports a registry setter or constructs a candidate registry. `UpdatePrices.stop()` clears the custom provider snapshot without touching unit state.

**JavaScript auto-update parses only v2 provider arrays.** _(implements "Phase 1 auto-updaters use only `data_v2.json`")_
In `packages/js/src/api.ts`, set `REMOTE_DATA_JSON_URL` to the raw `prices/data_v2.json` URL and narrow `ProviderDataValue` in `packages/js/src/types.ts` to `null | Provider[]`; remove `WrappedProviderData`. `setProviderData` accepts that value or a promise resolving to it. `activateProviderData(providers: Provider[]): Provider[]` calls `validateExtractorDestinations(providers, generatedRegistry)` before replacing and returning `providerData`; `null`, rejection, invalid arrays, and stale rejected promises retain the previous providers. Remove wrapped-payload parsing and registry activation.

**Python `Usage` and `ModelPrice` retain the completed direct-storage design.** _(implements "Manual Python pricing extensions remain supported", "Python `Usage` stores reported registry values directly", "Python `ModelPrice` uses direct public attribute storage", "Runtime provider parsing is an internal normalization step")_
In `packages/python/genai_prices/types.py`, `Usage` stores reported values in `__dict__`, filters them through bundled reported-key indexes for equality, representation, and addition, and performs explicit-only missing reads through bundled relationships. `Usage.from_raw(...)` reads attributes rather than mappings, ignores unknown source attributes for base pricing, and returns the registry-shaped copy while `ModelInfo.calc_price(...)` still passes the original usage to an overridden `ModelPrice.calc_price(...)`. Registered assignment stores or removes a reported value; unrelated names use ordinary object assignment.

`ModelPrice` remains a plain class with arbitrary price keyword arguments stored directly as public attributes. It has no metaclass, dataclass price fields, Pydantic core-schema customization, or `_extra_prices` mapping. `__getattr__` returns `None` only for absent bundled price keys; ordinary assignment and deletion remain unhooked. Effective-price collection excludes subclass-only custom state, and custom dataclass subclass constructors need not accept undeclared dynamic price keywords.

`_providers_from_raw(raw_providers: object) -> list[Provider]` is private runtime plumbing. It recursively normalizes flat and conditional price mappings, validates each price mapping as `dict[str, Decimal | TieredPrices | None]`, constructs `ModelPrice`, and then validates the surrounding provider dataclasses through a Pydantic adapter configured with `arbitrary_types_allowed=True`. Package-data generation and `UpdatePrices.fetch()` use this helper; provider-YAML schema generation continues using the build-time models.

**Python extractor construction retains its public validation boundary.** _(implements "Extractor destinations are externally reported bundled-registry usage keys", "Python extractor construction keeps its public compatibility boundary")_
In `packages/python/genai_prices/types.py`, `UsageExtractorMapping.dest: str` remains the destination field. `UsageExtractor.__post_init__(self) -> None` calls `_validate_usage_extractor_destinations(mappings: Sequence[UsageExtractorMapping]) -> None` against `_get_registry()` so direct custom construction rejects arbitrary names, price keys, and pricing-only destinations immediately. `UsageExtractor.extract(response_data: object) -> tuple[str | None, Usage]` accumulates counts by destination and returns registry-shaped reported usage without performing eager cross-field consistency validation.

**JavaScript retains open records and direct registry-aware usage reads.** _(implements "JavaScript retains plain usage and model-price objects", "Price data is complete while usage data may be incomplete", "`requests_kcount` remains an explicit one-request pricing exception")_
`packages/js/src/types.ts` retains `Usage = Record<string, number | undefined>`, `ModelPrice = Record<string, number | TieredPrices | undefined>`, and `UsageExtractorMapping.dest: string`. Extractor construction and provider activation validate every destination against `generatedRegistry.reportedUsageKeys`; extraction accumulates counts by usage key and returns a normalized plain `Usage`. `packages/js/src/usage.ts::normalizeUsage(...)` is used for extractor and normalized-return paths, while `packages/js/src/engine.ts::calcPrice(...)` reads the caller object directly. Missing-value safety, tier thresholds, one synthesized request, decomposition, and dimension-based aggregation match Python behavior.

**Both runtimes validate and price through one per-call structural collection.** _(implements "Phase 1 must avoid an unreasonable pricing hot-path regression", "Phase 1 removes repeated structural work from the pricing hot path", "JavaScript model validation materializes key and unit collections once")_
In `packages/python/genai_prices/types.py`, add `_collect_resolved_model_prices(model_price: ModelPrice, registry: UnitRegistry) -> tuple[tuple[UnitDef, Decimal | TieredPrices], ...]`. In `packages/js/src/engine.ts`, define `ResolvedPrice = Readonly<{ unit: UnitDef; price: number | TieredPrices }>` and `collectResolvedModelPrices(modelPrice: ModelPrice, registry: UnitRegistry): ResolvedPrice[]`; it forms one collection from the non-undefined `Object.entries(modelPrice)`. Each collector resolves registered units once; unknown base price keys raise, while subclass-only custom Python state remains excluded. Use this collection for:

- price-key, ancestor, and join validation;
- selecting non-`requests` decomposition units;
- determining whether any current price value is tiered;
- reading current price values;
- normalization and result aggregation.

`packages/python/genai_prices/validation.py` adds `validate_priced_units(priced_units: Collection[UnitDef], registry: UnitRegistry) -> None`, and `packages/js/src/validation.ts` adds `validatePricedUnits(pricedUnits: readonly UnitDef[], registry: UnitRegistry): void`. Existing key-based validation functions resolve keys once and delegate; the standard pricing functions call the resolved-unit form directly. They do not scan all registry units separately for key collection, unit collection, and tier detection.

**Decomposition stays direct and uncached.** _(implements "Every priced usage value lands in exactly one bucket", "Phase 1 deliberately excludes runtime-updated units and performance caches")_
Compute exclusive values from the resolved priced-unit collection and current explicit usage on every calculation. Do not add model-object caches, registry validation IDs, weak references, fingerprint state, prepared pricing plans, or decomposition coefficient caches.

**Authoring schemas and CLI remain registry-driven.** _(implements "Provider YAML authoring derives allowed keys from the registry", "CLI price presentation is registry-driven", "Phase 1 preserves the existing CLI surface")_
Provider YAML schema generation derives price-key and extractor-destination suggestions from the validated source registry. Repo-data tests enforce `_tokens`, `_mtok`, and the checked-in naming convention for direction, modality, and `token_type` combinations only for built-in token units; schema and runtime validation do not encode those token concepts.

In `packages/python/genai_prices/_cli.py`, `_collect_model_price_fields(results: Sequence[PriceCalculation]) -> list[str]` discovers stored/effective price keys, `_price_field_label(field_name: str) -> str` derives a display label from bundled unit dimensions and normalization, `_format_model_price_value(model_price: ModelPrice, field_name: str, *, use_color: bool) -> Text` formats one value generically, and `_format_model_prices(model_price: ModelPrice, *, split_lines: bool, use_color: bool) -> Text` combines the registered stored values. Existing CLI flags and option semantics remain compatible. Derived label wording follows current dimensions and does not add compatibility aliases or hardcoded branches when those dimensions change.

**Tests replace wrapped-payload lifecycle coverage with static-v2 coverage.** _(implements "Tests prove the complete releasable contract")_
Keep registry identity, normalization, public-name, interval/join closure, usage, decomposition, dynamic and misspelled price keys, modality prices, mixed aggregate output whose detail array omits the default text remainder, raw-provider normalization, schema, naming, CLI, custom-pricing, generated-output purity, and cross-runtime parity coverage. Include the `Usage.from_raw(...)` attribute-object boundary, ordinary unrelated usage attributes, direct `ModelPrice` storage without a side mapping, custom subclass behavior, flat and tiered dynamic prices, direct Python extractor-construction rejection, extractor inconsistency remaining lazy, existing CLI flags and output, no eager provider-activation model-price validation, and alignment with the shared examples. Replace tests for wrapped roots, candidate registry installation, registry rollback, and reset-on-stop with tests that:

- v1 artifacts are unchanged;
- v2 and its schema are provider-array artifacts with new registered keys;
- package unit data comes from the source registry rather than v2;
- successful and failed provider updates preserve the bundled registry identity;
- both auto-updaters point at v2;
- the hot path reuses one structural collection without behavior drift.
