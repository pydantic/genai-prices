# Data-Driven Unit Registry

**Code-level architecture is in [code-spec](code-spec.md).**

**The goal is to calculate prices as accurately as possible for a given request.**
This is the reason the system exists. Every design decision traces back to this: maximize pricing accuracy given the information available.

**Correct pricing semantics beat algorithmic convenience.** _(from "The goal is to calculate prices as accurately as possible")_
The decomposition algorithm is an implementation detail, not a contract users need to understand. A reasonable pricing shape must not produce strange results because of obscure Mobius-sign assumptions or because an intermediate unit happens not to exist. If a provider's commercial pricing says cached video tokens have a special price while the intermediate categories fall through to the broader input-token price, the registry and model price must express that explicitly: define the intermediate units and price them at the same numeric rate as the broader input unit. Repeating a price in an intermediate bucket is acceptable; silently returning a mathematically tidy but commercially wrong answer is not.

**Price data must be complete; usage data may be incomplete.** _(from "The goal is to calculate prices as accurately as possible")_
These are two different kinds of information with different completeness guarantees. Price data is authored by us (in provider YAML files) — we control it and can enforce completeness. If a model prices cached tokens and audio tokens, it must also price cached-audio tokens; anything less is a data quality bug, not an acceptable approximation. Usage data comes from callers at runtime — they provide what they have (sometimes just `input_tokens` and `output_tokens` from OpenTelemetry). We give the best answer possible with incomplete usage, but we don't accept incomplete prices.

**Every usage value must land in exactly one pricing bucket.** _(from "Price data must be complete")_
When a model prices overlapping units (e.g., "all input tokens" and "cached input tokens"), the system assigns each token-count to exactly one bucket. No double-counting, no dropped tokens. This is a correctness invariant enforced by price data completeness — it's not something we approximate or hope for.

**Validation exists to protect pricing semantics, not to satisfy the algorithm.** _(from "Correct pricing semantics beat algorithmic convenience", "Validation replaces what hardcoded fields gave us")_
Validation should reject data that is genuinely incomplete, ambiguous, or impossible to price accurately. Registry interval closure, registry join-closedness, price ancestor coverage, and price join coverage are semantic completeness rules, not arbitrary algorithmic hurdles. They may require explicit intermediate units or repeated numeric prices, but those definitions make the containment structure and fallback pricing unambiguous.

**Units are data, not code.**
A pricing unit — the thing you multiply usage by to get a cost — is defined once in a data file. That definition propagates to every language implementation. Adding a new unit requires editing a data file, nothing else. No Python class, no TypeScript interface, no schema update.

**Derive, don't duplicate.**
If information exists in one place, compute it elsewhere rather than copying it. Field names, validation rules, dimensional relationships — anything that can be derived from the registry should be derived, not restated in code.

**Backward compatibility is preserved unless clearly impractical.** _(from "Units are data, not code")_
Existing consumer code must continue to work. `model_price.input_mtok`, `usage.input_tokens`, `calc_price(usage)` — these patterns are the public API. This constrains how we implement data-driven units, not whether.

**All public API signatures are preserved.** _(from "Backward compatibility")_
`set_custom_snapshot(DataSnapshot | None)`, `UpdatePrices.fetch() -> DataSnapshot | None`, `calc_price()`, `ModelInfo.calc_price()` — all unchanged. `DataSnapshot` construction gains an optional `unit_registry` parameter (defaults to the global snapshot's unit registry), preserving backward compatibility. New behavior (validation at activation, global enforcement) is added without changing how these functions are called.

**No new code generation.** _(from "Backward compatibility", "Units are data, not code")_
Files that are currently hand-written stay hand-written. Generated code stays in files that are already generated (`data.py` for Python, the equivalent for JavaScript, JSON outputs). We do not introduce generation into `types.py`, `decompose.py`, `data_snapshot.py`, or any other hand-written module. Dynamic behavior (ModelPrice attribute access, Usage attribute access, extractor dest validation) comes from runtime registry lookups, not from generating source code with fields derived from the registry.

**Existing provider YAML files are mostly unchanged.** _(from "Backward compatibility", "Price data must be complete")_
Changes are justified only by genuine data gaps — e.g., a model that prices cached tokens and audio tokens but not cached-audio tokens fails a validation rule we're adding for good reason. No mass field renames, no restructuring for its own sake. This goes against "Price data must be complete" only in appearance — adding a missing cached-audio price IS making the data complete; it's not churn.

**Validation replaces what hardcoded fields gave us implicitly, and adds more.** _(from "Units are data, not code", "Derive, don't duplicate")_
Hardcoded ModelPrice fields provided implicit validation: a typo like `inptu_mtok` would fail because the field doesn't exist. Moving to data-driven units means explicit validation must replace that safety net and go further. Validation is comprehensive — it's cheaper to reject bad data at build/activation time than to debug wrong prices at runtime.

**Users can define custom units at runtime.** _(from "Units are data, not code")_
Without modifying the repository, without making a PR. Custom units are first-class: same mechanisms, same validation, same decomposition as built-in units. Users can add units to existing unit families or create entirely new unit families.

**Unit dimensions are unit-local.** _(from "Users can define custom units at runtime")_
Unit dimensions are defined on the units themselves; adding a unit with a new dimension key or value is how an existing family gains that axis or value.

**Runtime unit and price patches happen through `DataSnapshot`.** _(from "Users can define custom units at runtime")_
The public workflow should not require callers to manually copy the current registry, providers, models, or prices before making a small patch. A `DataSnapshot` is the user-facing editing surface: callers can edit a snapshot's unit registry and model prices directly through supported mutation APIs, then activate that snapshot if it is not already active. The implementation may copy internally to provide rollback, but that is not user-visible ceremony.

**Registry mutations commit only structurally valid states.** _(from "Users can define custom units at runtime", "Validation replaces what hardcoded fields gave us")_
Registry mutations validate usage key uniqueness, price key uniqueness, dimension-set uniqueness, interval closure, and join-closedness before the mutation becomes visible. There is no externally visible "build then validate" state for registry structure: mutation either commits a valid registry state or fails and leaves the registry unchanged.

**Unit definitions travel with prices, not just with the package.** _(from "Units are data, not code", "Users can define custom units at runtime")_
Currently, prices are in `data.json` which clients can auto-update at runtime (pulled on merge, before a package release). Unit definitions are bundled into the Python/JS packages separately from that runtime-updated payload, so they only change on package release. This means a new unit's prices could arrive before the client knows the unit exists. Unit definitions must be included in `data.json` (and `data_slim.json`) so they travel together with the prices that depend on them. When a client pulls fresh price data, it gets the units too.

**`UnitRegistry` owns the runtime unit graph.** _(from "Units are data, not code", "Users can define custom units at runtime")_
A single class that owns all unit families and their units.

**`UnitRegistry` construction parses and indexes raw unit data.** _(from "`UnitRegistry` owns the runtime unit graph")_
Constructed from a raw dict (the `unit_families` section of `data.json` or `units.yml`), it parses, validates structural integrity (usage-key/price-key/dimension-set uniqueness, interval closure, join-closedness), builds flat indexes for O(1) lookup by usage key and O(1) price-key resolution, and fills in back-references so each unit knows its usage key, its price key, its family ID, and has a direct reference to its family object.

**`UnitRegistry` mutation keeps indexes and relationships current.** _(from "`UnitRegistry` owns the runtime unit graph", "Runtime unit and price patches happen through `DataSnapshot`")_
`UnitRegistry` is mutable: adding a family or adding units to an existing family validates and updates the indexes. These mutations may be exposed directly on `UnitRegistry` or as convenience methods on `DataSnapshot`, but callers should not need to construct a parallel registry by hand for ordinary patches.

**Unit definition objects have one runtime model per concept.** _(from "`UnitRegistry` construction parses and indexes raw unit data")_
`UnitDef` and `UnitFamily` are dataclasses (not Pydantic models) — the registry constructs them with all fields populated, including back-references. There is one model class per concept, not a "raw" and "parsed" variant. In the YAML/JSON, usage keys are dict keys; the registry promotes them to fields on construction, and `price_key` defaults to the usage key when omitted.

**Existing-family unit edits are batch operations.** _(from "Users can define custom units at runtime", "`UnitRegistry` mutation keeps indexes and relationships current")_
The registry does not expose a separate "add exactly one unit" operation for existing families. It exposes one batch operation that accepts one or more unit definitions for a single existing family, applies them to a candidate family, validates the complete candidate registry, and commits only if the final state is valid. Passing a one-unit batch is how callers perform a simple addition. Passing a larger batch is how callers add units that are only valid together under structural rules such as interval closure or join-closedness. Because dimensions are unit-local, there is no separate operation for adding dimension values or whole dimension axes; a staged unit that uses `{modality: video}` or `{region: us}` introduces that value or axis if the final registry validates. If validation fails, the original registry is unchanged.

The batch operation does not invent or require a shape. It does not auto-generate units for every dimension value, require a new modality to match existing modalities, add prices, validate model prices, update extractors, or activate a `DataSnapshot`. It only edits unit definitions. After editing units, callers add matching provider/model price changes to the same snapshot. If that snapshot is inactive, `set_custom_snapshot` performs price-level and extractor validation before activation. If that snapshot is already active, supported price mutations must invalidate stale known-valid/decomposition state and either validate the changed model price immediately or leave it to the one-model defensive validation in `calc_price`; in either case, unchanged trusted prices must not be revalidated just because the snapshot received one patch.

**Runtime unit patches use the same batch boundary in every language.** _(from "Existing-family unit edits are batch operations")_
The concrete method name can be language-idiomatic, but the shape is settled: pass a batch of unit definitions for an existing family and validate once after the batch is staged. Python might accept a mapping keyed by usage key, while JavaScript might accept an array of objects with `usageKey`; either representation is the same operation. A hypothetical future `video` modality in the `tokens` family must be possible, as must other future dimensions in that family, without forcing callers to create an unrelated new family or rely on invalid intermediate registry states.

**Unit families live in the data snapshot alongside prices.** _(from "Users can define custom units at runtime", "Unit definitions travel with prices")_
`DataSnapshot` carries a `UnitRegistry`, just as it carries providers and models. Construction accepts an optional `unit_registry` parameter — when omitted, it defaults to the current global snapshot's unit registry, preserving backward compatibility for existing code that constructs a `DataSnapshot` from a providers list. The bundled snapshot includes a unit registry built from generated code; `data.json` auto-updates include unit families too. Adding a custom unit or price means editing the relevant snapshot through supported mutation APIs. Those APIs may copy internally for rollback, but the public workflow should be "edit the snapshot, then activate it if needed," not "manually copy the registry and provider tree, patch the copies, then assemble a new snapshot." `UnitRegistry`, `UnitFamily`, and `UnitDef` are public classes.

**There is one global DataSnapshot.** _(from "Unit families live in the data snapshot")_
The library has one active snapshot at any time — either the bundled default or a custom one set via `set_custom_snapshot`. This is already the de facto design; we make it explicit. Price calculation and usage extraction read the unit registry from the global snapshot. There is no need to thread the registry through method parameters or store it on individual objects. Global state is the natural model for a pricing database — there is one set of prices, one set of units, one answer to "what does this model cost."

**Non-global snapshots are staging objects, not execution contexts.** _(from "There is one global DataSnapshot", "Validation is split between the registry and `set_custom_snapshot`")_
A `DataSnapshot` can exist before activation so callers can fetch data, inspect models, and patch units or prices in place. But until `set_custom_snapshot(snapshot)` succeeds, that snapshot is not treated as authoritative for pricing or extraction. This is deliberate. `ModelPrice` and extractor objects are constructed before activation and do not validate their unit references at construction time; the only place the library checks "do these model price keys resolve to real units, and do these extractor destinations name real usage keys?" for inactive snapshots is snapshot activation. Allowing arbitrary inactive snapshots to execute `calc` or `extract_usage` would blur the line between staged and validated state and encourage reliance on snapshots whose prices/extractors have not yet been globally accepted.

**Validation is split between the registry and `set_custom_snapshot`.** _(from "There is one global DataSnapshot", "Validation replaces what hardcoded fields gave us")_
Structural validation — usage key uniqueness, price key uniqueness, dimension-set uniqueness, interval closure, and join-closedness — happens in the `UnitRegistry` on construction and mutation. The registry is always structurally sound; you cannot put it into an inconsistent state. Price-level validation — model price keys are registered price keys, ancestor coverage, join coverage — happens at `set_custom_snapshot` time when a model price is not already known valid for that registry and price-key set. Price keys resolve to usage keys before ancestor and join checks. Runtime validation should iterate each model's priced keys and use registry indexes or cached relationship helpers; it should not scan the full registry for every model if a straightforward indexed check is available.

Known-valid state and cached decomposition state are separate. A `ModelPrice` can be known valid for a registry and price-key fingerprint without having precomputed decomposition coefficients or a cached decomposition plan. Known-valid state proves price-level validation has already happened, either in the repo build pipeline or in a previous runtime validation. A cached decomposition plan is an optional optimization for `calc_price`, not a correctness requirement. Both kinds of state are runtime-private and scoped to the registry plus current price-key fingerprint; neither is serialized into `data.json`, `data_slim.json`, generated Python data, or generated JavaScript data. If validation fails, the previous global snapshot remains in place.

**ModelPrice construction does not validate against the registry.** _(from "Validation is split between the registry and `set_custom_snapshot`", "Users can define custom units at runtime")_
No price-key lookup, no ancestor/join coverage checks at ModelPrice `__init__` time. This allows constructing ModelPrice objects with custom price keys before the snapshot containing those unit definitions is assembled and activated. Validation happens later, when the full context is available. Repo-built generated model prices and official auto-update payloads are trusted because the repo build pipeline already validated them; loading them must not run price-level validation for every model. Generated artifacts still contain only raw unit and price data, not decomposition caches or bulky validation metadata. `calc_price` resolves stored price keys to usage keys and looks up unit metadata (family, dimensions, usage keys) from the global snapshot via `get_snapshot()`. If the model price is already known valid, `calc_price` can compute decomposition directly or build/cache a plan without rerunning price-level validation. If it is not known valid, `calc_price` validates that one model price once, marks it known valid, then prices it. Standalone ModelPrice objects that bypass `set_custom_snapshot` are not a supported pattern, but this defensive check prevents silent wrong pricing in ordinary calculation paths when validation was accidentally skipped.

**`calc_price` is a hot path — keep it simple and avoid unnecessary heavy work.** _(from "ModelPrice construction does not validate against the registry", "Expensive validation happens once at activation time")_
`calc_price` runs after every LLM call, so correctness should not come from re-running validation or searching all models. Its required validation work is a cheap known-valid check: if the model price is known valid for the active registry validation id and the same price-key fingerprint, no price-level validation runs. If known-valid state is missing or stale, only that one model price is validated and marked. Decomposition coefficients, if cached, are per model price rather than global to the registry, because only that model's priced units participate in decomposition. They are independent of the usage values supplied on a particular call, so they can be reused across calls until the registry or model price-key set changes. Caching those coefficients is a useful optimization if it stays simple, but the implementation may compute them during `calc_price` if caching proves awkward. Price key → usage key and usage key → unit definition are direct dict lookups via the registry's flat indexes; each `UnitDef` carries a direct reference to its `UnitFamily`, so `unit.family.per` requires no second lookup. Raw usage wrapping may scan registered usage keys when reading an arbitrary object or Mapping — this preserves the permissive public API and is acceptable while the registry is small. Once usage is wrapped in the registry-aware usage representation, `calc_price` asks for only the values it needs, decomposes the model's priced units, checks contradictory or non-interpretable usage only when it affects those priced units, and sums costs in O(k) or O(k²) in the number of priced units for that model (typically 2–10), not in the size of the registry. Tier selection reads the total input-token value but does not audit descendant counts when that total was reported. Optimize the easy parts now: use flat indexes, avoid full-registry scans in decomposition, and keep the design open to decomposition relationship indexes later if profiling shows a need.

If the same `ModelPrice` object appears in multiple snapshots, its known-valid state and any cached decomposition state are reusable only when the active registry validation id and price-key fingerprint match. A snapshot with deleted or changed units has a different registry validation id, so those states are stale and validation runs before pricing unless the registry mutation path can prove that the old validation remains valid for unchanged prices.

**`calc` and `extract_usage` on DataSnapshot require it to be the current global.** _(from "There is one global DataSnapshot")_
These methods raise an error if called on a non-global snapshot. This prevents price calculations and usage extraction against a snapshot whose registry isn't the active one — the unit registry for decomposition and validation always comes from the global snapshot. More importantly, it preserves the staged-vs-active boundary: inactive snapshots may carry a registry, but they have not yet earned the guarantee that all enclosed model price keys and extractor destinations are valid against it. Lookup-only methods are allowed on inactive snapshots; execution methods are not. `ModelInfo.calc_price()` itself does not try to prove where the `ModelInfo` came from; calling it on a model object obtained from an inactive snapshot is unsupported but allowed, and it uses the active global registry like any other pricing call.

**`find_provider_model` works on any snapshot, global or not.** _(from "There is one global DataSnapshot", "Backward compatibility")_
`find_provider_model` and `find_provider` are pure lookups — name/pattern matching that does not need the unit registry. They must work on non-global snapshots because the `UpdatePrices.fetch()` customization pattern requires it: construct a snapshot from fetched data, query it to find existing models, patch or add custom models on that snapshot, return the snapshot for the framework to activate.

**The system is general across unit families.** _(from "Units are data, not code")_
Any unit with the structure `usage_value x price / normalization_factor` is expressible: tokens (per million), requests (per thousand), characters, duration, etc. Tokens are the first and most complex family (because of overlapping usage), but nothing in the system is token-specific. Future unit families require only a data file edit.

**A unit family groups units whose usage values can overlap.** _(from "The system is general")_
Each family has a normalization factor (`per: 1_000_000` for tokens). Within a family, a more-specific unit's usage is a subset of a less-specific one's (`cache_read_tokens` is a subset of `input_tokens`). Between unit families, there is no overlap — request counts don't interact with token counts.

**`requests_kcount` becomes the price key for a unit in a `requests` family.** _(from "The system is general", "A unit family groups units")_
The existing `requests_kcount` field is a hardcoded special case — a price per thousand requests. It belongs in the registry as unit `requests` in a `requests` family with `per: 1_000` and `price_key: requests_kcount`. The `requests` family has no dimensions — it's a single unit with no overlapping subtypes (decomposition is trivial — the formula returns the usage value directly). Model prices continue to use `requests_kcount`; usage continues to use `requests`. This makes request pricing subject to the same mechanisms as token pricing — validation, schema autocomplete, and the same code path in `calc_price`. When a priced model includes the requests family and the caller did not supply `requests`, `calc_price` treats the call as `requests=1`. This default lives in pricing, not in `Usage`: `Usage` models supplied or inferred usage data, while "one API call per `calc_price` invocation" is a backward-compat pricing rule. This preserves backward compatibility with existing code that doesn't pass request counts.

**Dimensions define unit specificity and overlap.** _(from "A unit family groups units")_
Each unit carries zero or more categorical dimension assignments: `{direction: input}`, `{direction: input, cache: read}`, `{direction: input, modality: audio, cache: read}`. More dimensions = more specific. Dimension keys and values are not declared separately on the family; they are derived from the units that use them. The containment relationship — which units are ancestors/descendants of which — is determined by set inclusion on dimensions: if A's dimensions are a subset of B's, A is an ancestor of B. This structure is the basis for decomposition and for all validation rules.

**Decomposition uses dimensions, not hardcoded subtraction chains.** _(from "Dimensions define unit specificity", "Every usage value must land in exactly one pricing bucket")_
The current code subtracts specific unit values from general ones in a manually maintained order. The replacement: Mobius inversion on the containment poset defined by dimensions. The algorithm takes the set of priced units, computes each one's "leaf value" (exclusive portion of usage), and guarantees no double-counting. No code names specific units — the algorithm works from the dimension structure alone. See [algorithm](algorithm.md) for the formula and multi-way overlap handling.

**Only priced units participate in decomposition.** _(from "Decomposition uses dimensions")_
If a model doesn't define price key `input_audio_mtok` for unit `input_audio_tokens`, audio tokens remain part of the `input_tokens` catch-all priced by `input_mtok` — they are not carved out. The decomposition is determined by the set of units that have prices for the current model, not the full registry. This is what makes partial pricing work: a model that only prices `input_mtok` and `output_mtok` still produces correct results.

**Decomposition operates within a family.** _(from "Decomposition uses dimensions", "A unit family groups units")_
Token decomposition does not affect request pricing. Each family's decomposition is independent — there is no cross-family overlap to resolve. Price calculation starts from the model's price keys — each resolves to exactly one usage-keyed unit, which belongs to exactly one family. Group by family, decompose each group, price each leaf. No exhaustive scan of the registry's unit families is needed.

**Costs are aggregable by dimension filter.** _(from "Decomposition uses dimensions", "Dimensions define unit specificity")_
After decomposition, the cost for each priced unit is its leaf value times its price. These per-unit costs can be summed by dimension filter: "total cost for {direction: input}" sums the leaf costs of all priced units whose dimensions are a superset of the filter. This is the general mechanism — `input_price` and `output_price` are specific instances of it.

**`input_price` and `output_price` are backward-compat accessors over direction-filtered costs.** _(from "Costs are aggregable by dimension filter", "Backward compatibility")_
`calc_price` currently returns `{input_price, output_price, total_price}`. The input/output grouping is a cost aggregate filtered by `{direction: input}` and `{direction: output}` respectively. These names are kept for backward compatibility. Costs from unit families without a `direction` dimension (e.g., requests) appear only in `total_price`, not in `input_price` or `output_price`.

**Nice-to-have: aggregate usage count queries by dimension filter.** _(from "Decomposition uses dimensions", "Dimensions define unit specificity")_
Given a usage object and a dimension filter, it should be possible to ask "how many tokens match `{direction: input}`?" — by summing the leaf values of all priced units whose dimensions are a superset of the filter. Unlike cost aggregation, this is a usage-and-dimensions operation that doesn't involve prices. Examples: all input tokens ({direction: input}), all cache read tokens ({cache: read}), all audio tokens ({modality: audio}) regardless of direction, all tokens (empty filter). A filter matching no priced units returns zero. The API is not yet defined — it could be a standalone function that accepts a usage object and a filter. This is not required for the initial implementation but the dimension system makes it natural.

**TieredPrices is not refactored in this change.** _(from "Backward compatibility")_
The existing TieredPrices mechanism (threshold-based pricing where crossing a tier applies that rate to all tokens) continues to work as-is. The tier threshold is determined by the total input token count — the provided or inferable usage for the catch-all `input_tokens` unit, whose price key is `input_mtok`. If `input_tokens` is reported, tier selection uses that value directly and does not reconcile it against more-specific usage fields. If `input_tokens` is absent and the supplied usage does not determine a coherent total, price calculation raises instead of guessing a tier. The aggregate query concept generalizes this into a dimension filter ({direction: input}), but TieredPrices is not modified to use it — that's a separate future change.

**Unspecified dimensions mean catch-all: the unit prices whatever isn't claimed by a more specific unit.** _(from "Dimensions define unit specificity", "Only priced units participate")_
Unit `input_tokens` (price key `input_mtok`) has no `modality` dimension — it's the catch-all for all input tokens not claimed by a modality-specific unit. The catch-all implicitly functions as the default modality. For text-primary models (most models), `input_mtok` IS the text price — no `input_text_mtok` needed. For image-primary models, the catch-all should be the image price: set `input_mtok` and `input_image_mtok` to the same value, and define `input_text_mtok` as the exception. See [examples](examples.md) for concrete pricing patterns.

**Ancestor coverage: pricing a unit requires pricing all its ancestors within the same family.** _(from "Price data must be complete", "Decomposition uses dimensions")_
If a model prices unit `cache_read_tokens` via `cache_read_mtok` ({direction: input, cache: read}), it must also price ancestor unit `input_tokens` via `input_mtok` ({direction: input}) — both are in the tokens family. Without the ancestor, usage reported at the general level (just `input_tokens`, no breakdown) would have no price. This is a price data completeness requirement — validated, not assumed.

**Join coverage: pricing two overlapping units within the same family requires pricing their intersection.** _(from "Price data must be complete", "Decomposition uses dimensions")_
If a model prices both `cache_read_mtok` (unit `cache_read_tokens`) and `input_audio_mtok` (unit `input_audio_tokens`), their join — unit `cache_audio_read_tokens` with the union of both dimension sets, `{direction: input, cache: read, modality: audio}` — must also be priced. Registry join-closedness guarantees that join unit exists when the two priced units are compatible. Without the join price, Mobius inversion double-counts tokens that belong to both parents. This is not a nice-to-have — it's a direct consequence of requiring accurate prices. Each token in one bucket, not two.

**Decomposition correctness depends on registry closure and price coverage.** _(from "Only priced units participate", "Ancestor coverage", "Join coverage")_
The Mobius inversion formula sums only over priced units, using sign `(-1)^(depth(V) - depth(U))` from the full poset. This gives correct leaf values only when the registry and the model's price set have no structural gaps. Registry interval closure makes sure comparable unit chains have their intermediate units. Price ancestor coverage then makes sure those intermediate units are priced when a model prices the more-specific unit. Registry join-closedness makes sure overlapping registered units have an explicit intersection unit. Price join coverage then makes sure that intersection is priced when a model prices both overlapping parents. The validation rules are not just data quality checks — they are preconditions for the algorithm's correctness.

**Registry interval closure: comparable unit pairs must have their intermediate units.** _(from "Decomposition correctness depends on registry closure and price coverage", "Users can define custom units at runtime")_
For every comparable pair of registered units `A <= B` in the same family, every dimension set strictly between `A.dimensions` and `B.dimensions` must also correspond to a registered unit in that family. "Between" means: start with `A.dimensions` and add any non-empty proper subset of the dimension assignments that are present on `B` but absent from `A`. This rule is local to existing comparable units; it does not require standalone nonsensical units such as `{cache: read}` unless another registered comparable pair makes that dimension set an actual intermediate.

Example: if a runtime extension wants to add:

```text
cache_video_read_tokens   {direction: input, cache: read, modality: video}
```

and the family already contains:

```text
input_tokens              {direction: input}
```

then interval closure requires both intermediate units:

```text
cache_read_tokens         {direction: input, cache: read}
input_video_tokens        {direction: input, modality: video}
```

If cached video has a special price but ordinary video input should use the normal input-token price, the model expresses that with an explicit intermediate price:

```text
input_mtok
cache_read_mtok = same numeric value as input_mtok, if cached reads otherwise fall through
input_video_mtok = same numeric value as input_mtok
cache_video_read_mtok
```

That repeated price is acceptable. The alternative — omitting `input_video_tokens` and letting the full-depth formula add `cache_video_read_tokens` back into the input catch-all — would be silently wrong.

**Registry interval closure, registry join-closedness, and price join coverage are cumulative.** _(from "Registry interval closure", "Registry join-closedness", "Join coverage")_
These rules are not alternatives and they are not redundant. Interval closure prevents units from skipping comparable intermediates. Registry join-closedness prevents overlapping unit concepts from lacking an explicit intersection unit. Price join coverage prevents a particular model's overlapping prices from double-counting usage by requiring the join price too. Keeping all three does not block meaningful pricing shapes, but it can require explicit intermediate units and repeated numeric prices. That is the intended tradeoff: the data states the fallback rate directly instead of relying on sparse-registry fallthrough through a missing unit.

**The registry defines units symmetrically across modalities.** _(from "Registry join-closedness")_
For the built-in `tokens` family, each modality gets the same four unit patterns: input, output, cache read, and cache write — even where no provider currently uses them. This is not the full Cartesian product of all dimension values (nonsensical combinations like `{direction: output, cache: read}` are excluded since caching is input-side). The built-in symmetry is a registry-data choice for today's token modalities, not a general validation rule. Custom unit families and runtime extensions do not need to give every value on a dimension key the same shape; they only need to satisfy the structural rules that apply to the units they actually define.

**Registry join-closedness: compatible unit pairs must have their join in the family.** _(from "Decomposition correctness depends on registry closure and price coverage", "Users can define custom units at runtime")_
Two units are compatible when they share no dimension key with conflicting values — `{direction: input, cache: read}` and `{direction: input, modality: audio}` are compatible; `{direction: input}` and `{direction: output}` are not. For every compatible pair within a family, the unit with the union of both dimension sets must exist in that family. Without this, Möbius inversion silently misallocates usage between overlapping buckets — the overlap has no unit to land in, so the formula distributes it incorrectly across parents. This is a structural requirement on the registry itself, independent of any model's prices. For built-in unit families, symmetric definitions satisfy it by construction. For custom unit families, it is validated by the `UnitRegistry` on construction and mutation — you cannot create or mutate a registry into a non-join-closed state. Existing-family unit additions are batched, so a group that is only valid together is added in one operation and validated once at the end. The registry does not require `video` to have the same shape as `audio`, `image`, or `text`.

**`data.json` becomes a top-level dict, not a bare list.** _(from "Unit definitions travel with prices")_
Currently `data.json` is a bare JSON array of providers. Adding unit families requires a top-level wrapper: `{"unit_families": {...}, "providers": [...]}`. This is a one-time schema break — old clients that validate the fetched JSON as a list of providers will fail. The top-level should have been a dict from the start; this change enables future extensibility without further breaks. After this, adding new top-level fields (e.g., metadata) won't require another schema change. `data_slim.json` undergoes the same structural change — both files carry unit families. The slim variant continues to omit provider descriptive fields and free models, as it does today; unit family data keeps the runtime fields needed to parse the registry.

**The registry is a YAML file that defines all built-in units.** _(from "Units are data, not code", "Derive, don't duplicate")_
One file, checked into the repo (`prices/units.yml`). It defines unit families, their normalization factors, and their units. Units are keyed by usage key; each unit may define a separate price key and always defines dimension assignments. The built-in `tokens` units use dimension keys such as `direction`, `modality`, and `cache`, but the family does not separately declare the possible keys or values. At build time, this file is compiled into `data.json` alongside prices. At runtime, the registry is loaded from the same data the client already has — no separate file needed.

**Naming optimizes for writability: people discover names by pattern-matching from existing ones.** _(from "Backward compatibility")_
Provider YAML files and usage objects are written frequently. People figure out names by looking at what's already there and extrapolating, not by memorizing a convention document. Consistency across unit names matters because inconsistency is how people get tripped up — seeing `cache_audio_read_mtok` and guessing `cache_image_read_mtok` must work (and it does, because the names are consistent). The convention for cached+modality units follows the pre-existing `cache_audio_read` pattern: `cache_{modality}_{op}`. For non-cached: `{direction}_{modality}`. Token price keys use the same pattern with `_mtok`; token usage keys use the same pattern with `_tokens`.

**Generated JSON schemas provide editor autocomplete for provider YAML files.** _(from "Naming optimizes for writability", "The registry is a YAML file")_
The unit registry generates a JSON schema that provider YAML files reference (via the `yaml-language-server` directive). This gives autocomplete and validation in editors for both price keys and extractor destination fields, reducing reliance on memorizing names. The schema is regenerated whenever the registry changes.

**ModelPrice supports attribute access backed by registry data.** _(from "The registry is a YAML file", "Backward compatibility")_
ModelPrice is not a plain dict — that would break `model_price.input_mtok`. It's an object that supports attribute access, but the set of valid attributes comes from the registry's price keys. New units added to the registry are immediately accessible as attributes when they define new price keys. Legacy names like `input_mtok` continue to work. Keeping typed property definitions for existing names is acceptable as a backward-compat shim — but it's a convenience for type checkers, not the source of truth. Future names need no code changes. Supported model-price mutation paths that add or remove effective price keys invalidate private known-valid state and any cached decomposition state for that model price. They do not need to revalidate immediately; `set_custom_snapshot` validates stale or untrusted model prices early, and `calc_price` performs one-model validation if a mutated price is used before activation.

**Custom price patches work by mutating staged snapshot data and activating it.** _(from "Users can define custom units at runtime", "ModelPrice supports attribute access backed by registry data")_
If a user finds that bundled prices are missing a price key they need — for example, cache-token prices — they can edit the relevant `ModelPrice` on a snapshot and activate that snapshot if it is not already active. If the patch also needs new units, they edit the unit registry on that same snapshot. Supported mutations clear known-valid state and any cached decomposition state only for the changed `ModelPrice` objects. Unchanged model prices from trusted bundled or auto-updated data keep their known-valid provenance when the active registry is unchanged. `set_custom_snapshot` then validates the missing/stale model prices against the staged snapshot's registry before activation, records known-valid state again only after the full snapshot is valid, and does not revalidate every trusted built-in price just because it shares a snapshot with one custom addition. It may also compute/cache decomposition for the changed model prices while validating them, but that is optional and it must not compute plans for every unchanged built-in price eagerly. If a user mutates a `ModelPrice` object directly and calls `calc_price` without activating a new snapshot, that direct mutation is still detected by the missing/stale known-valid state and the one-model defensive validation runs before pricing.

**`Usage` is a registry-aware class that infers and serves correct values.** _(from "The registry is a YAML file", "Derive, don't duplicate")_
Each unit is keyed by its usage key — the attribute name to look up on the usage object (e.g., `input_tokens`). The corresponding price key (e.g., `input_mtok`) is only the model-price/provider-YAML alias. The `Usage` class has no hardcoded fields — it uses the global unit registry to understand the relationship between usage keys. Construction via keyword arguments for known usage keys (`Usage(input_tokens=1000)`) or from an arbitrary object (`Usage.from_raw(obj)`, extracting known usage keys via `getattr`/Mapping access) continues to work. Construction stores the reported values and does not attempt to prove that they are mutually consistent. This is intentional: real provider APIs can return contradictory usage, and `extract_usage` must be able to report "this is what the API said" without failing as the messenger. `Usage` supports keys from any family — token counts and request counts coexist on the same object.

**`Usage` infers ancestor values from descendants.** _(from "Usage is a registry-aware class", "Price data must be complete; usage data may be incomplete")_
If a caller provides `Usage(input_audio_tokens=300, output_tokens=500)` without `input_tokens`, `usage.input_tokens` returns 300 — inferred from the descendant. Reported values are never overwritten by inference. Inference is lazy: construction stores reported values only, and registered attribute access first returns a stored value without checking whether other stored values contradict it. Only missing-value reads attempt inference from the currently stored descendants. Do not cache inferred usage values; recompute them when requested. This avoids subtle bugs where derived ancestors start behaving like reported data. Accessing a usage key that exists in the registry but has no data and no relevant descendants returns zero.

Not every missing ancestor can be inferred. If available values cross independent dimensions without the totals or intersections needed to determine a unique answer — for example modality counts plus cache counts but no overall `input_tokens` or cached-by-modality intersections — `usage.input_tokens` raises a user-facing error instead of choosing an arbitrary axis. If a reported value contradicts more-specific reported values, the `Usage` object can still exist and direct reads of stored fields still succeed, but any missing-value inference or price calculation that needs to reconcile the contradiction raises. `Usage.__add__` (streaming accumulation) sums reported values only; derived ancestors are recomputed lazily when requested.

Open implementation point: `Usage` does not currently need to remember whether each value was reported or inferred. With lazy inference, direct reads of stored values are inert, missing-value inference knows it is computing a missing value, and pricing errors can be explained from the stored values involved in the contradiction. Keep the API and internal state minimal. If implementation work finds a concrete diagnostic that genuinely needs per-value provenance, add private bookkeeping then; otherwise do not store it.

**Decomposition stays in `calc_price`, not on `Usage`.** _(from "Usage is a registry-aware class", "Decomposition uses dimensions")_
Decomposition — computing how much of a usage value is exclusive to each priced unit (Mobius inversion) — depends on the set of priced units, which is pricing data. `Usage` does not expose a leaf-value method. `calc_price` handles the Mobius inversion internally, reading values from the smart `Usage` as needed. This keeps `Usage` focused on what it naturally owns: reported values and value inference. `calc_price` owns the price-dependent decomposition. TieredPrices asks `usage.input_tokens` for the total input count; if that value was reported, it is used directly. If it was not reported and cannot be inferred coherently, price calculation fails with a usage error instead of selecting a nonsensical tier.

**The usage parameter type is `object`, not a library-specific type.** _(from "Usage is a registry-aware class")_
`calc_price` accepts any object as usage. Callers are not required to use the library's `Usage` class — they can pass their own dataclass, a namespace object, a dict, or anything else. Internally, `calc_price` wraps non-`Usage` objects in `Usage.from_raw(obj)`, which extracts known usage keys via `getattr`/Mapping access and produces a smart `Usage` that can infer missing values lazily. All downstream decomposition reads from the smart `Usage`. This means callers get full inference and correct decomposition regardless of what type they pass — but callers who construct `Usage` directly benefit from the same lazy inference when they access registered attributes. `AbstractUsage` remains as a public export (assigned to `object`) solely for backward compatibility. Internally, the library does not use `AbstractUsage` in any signature — it uses `object` directly, because that is what it actually means.

**The extraction pipeline is data-driven end-to-end.** _(from "Units are data, not code", "Usage is a registry-aware class", "No new code generation")_
Extractors are already defined in provider YAML — they map paths in API responses to usage fields. Currently the output side is hardcoded: `UsageExtractorMapping.dest` is a `UsageField` literal type, and the `Usage` object has fixed fields. Both constraints are removed. `dest` becomes a plain string referencing any usage key in the registry. The `UsageField` literal type goes away. The generated JSON schema for provider YAML files validates extractor `dest` fields against the registry's usage keys, giving autocomplete when writing extractors. At runtime, extractor `dest` values are validated against the registry. The full chain — provider YAML extractor definition → `Usage` object → `calc_price` — requires no code changes when new units are added to the registry. A new unit with a new usage key is immediately available as an extractor destination, a `Usage` attribute, and a pricing input.

Extraction reports provider data; it does not certify that the provider's usage numbers are internally coherent. If an API response contains contradictory registered usage fields, `extract_usage` still returns a `Usage` object containing those values. Direct reads of fields that the provider supplied return the supplied values. The contradiction becomes a hard error only when code asks for a missing value whose inference cannot be determined, or when `calc_price` must reconcile the contradictory values to price a unit the model actually prices.

**Price key defaults to usage key if not specified.** _(from "Usage is a registry-aware class")_
Usage keys like `input_tokens` name the raw count the caller provides. Price keys like `input_mtok` encode the pricing field name used in provider YAML and on `ModelPrice`, including normalization hints from the old API. The `price_key` field bridges the two — the system looks up `input_tokens` on the usage object and applies the `input_mtok` price with the family's `per: 1_000_000`. For the tokens family, every unit has an explicit `price_key` because of this naming split. For unit families where the usage key and price key are the same (e.g., a future `tool_calls` family), the default avoids redundancy.

**Incomplete usage is handled gracefully, not rejected.** _(from "Price data must be complete; usage data may be incomplete", "Usage is a registry-aware class")_
A caller with only `{input_tokens: 1000, output_tokens: 500}` gets a valid price at catch-all rates — and if the model prices requests, `calc_price` counts one request automatically. A caller with only `{input_audio_tokens: 300}` gets a valid price at the audio rate — `Usage` can infer `input_tokens=300`, and the leaf value for unit `input_tokens` is zero (all input tokens are accounted for by the audio descendant). A caller with detailed breakdowns gets a more precise price. If a caller provides a usage key (e.g., `input_video_tokens: 100`) for a model that doesn't price unit `input_video_tokens` via `input_video_mtok`, those tokens silently remain in the catch-all ancestor when the necessary ancestor totals are known or inferable — only priced units become cost buckets. Unpriced reported usage keys are ignored unless they are needed to infer a missing priced value or a missing tier threshold. For example, if a model only has an input catch-all price and the caller reports `{input_tokens: 100, cache_read_tokens: 200}`, price calculation can use `input_tokens=100` and ignore the impossible unpriced cache count, including for tier selection. If the model also prices `cache_read_tokens`, the same usage must fail because the priced buckets cannot be reconciled. Usage for a family with no priced units at all (e.g., `requests: 5` on a model with no request pricing) is silently ignored — there is nothing to decompose or price.

**Raw usage objects remain permissive; unknown fields are ignored.** _(from "Incomplete usage is handled gracefully", "Backward compatibility")_
When usage arrives as an arbitrary object or Mapping, the library reads the known usage keys and ignores extra attributes/keys. This preserves the existing contract in both Python and JavaScript, where callers may attach unrelated fields to the same object. It also means typos in raw usage objects are not caught by runtime validation in this change. Tightening that contract would be a separate API decision, not part of the unit-registry work.

**Inconsistent usage is allowed to exist but rejected when interpreted.** _(from "Every usage value must land in exactly one pricing bucket", "Incomplete usage is handled gracefully")_
Incomplete and inconsistent are different. Missing usage is handled by lazy inference in `Usage` when a value is requested. Inconsistent usage is allowed to be constructed and extracted, because provider APIs can report contradictory values and extraction should not fail just for relaying them. But when reported descendant counts cannot fit within reported ancestor counts after accounting for overlaps — e.g., `{input_tokens: 100, cache_read_tokens: 200}` — that's contradictory: a subset can't exceed its superset. The error is demand-driven: reading `usage.input_tokens` returns `100` and reading `usage.cache_read_tokens` returns `200` because both fields were stored from the reported usage; ordinary direct `Usage` reads do not complain. Asking for some missing inferred aggregate that depends on this relationship can raise. `calc_price` raises only when it must reconcile the impossible relationship to compute a priced bucket; it does not proactively validate contradictory unpriced usage values that are irrelevant to the model's prices, and tier selection uses a reported `input_tokens` total as-is. Negative buckets should trace back to the contradictory stored values, not to an inference gap; if an impossible relationship involves a value computed while answering a missing-field request, the message should describe the stored values that made the requested inference impossible.

**Error messages describe the data problem, not the algorithm.** _(from "Inconsistent usage is allowed to exist but rejected when interpreted", "Validation replaces what hardcoded fields gave us")_
Users see error messages, not source code. No error should mention leaves, Mobius inversion, posets, depth, coefficients, or dimensions. A negative leaf means more-specific counts do not fit inside a less-specific count after overlap correction — say that: "cache_read_tokens (200) exceeds input_tokens (100) — cached input tokens are a subset of input tokens." When a missing-field inference fails, say that the requested value could not be inferred from the stored values; do not present computed intermediate values as if the caller supplied them. The explanation is derived from the registry structure (which units are more/less specific), not hardcoded per unit pair. Validation errors (ancestor coverage, join coverage, invalid keys) follow the same principle: describe what's wrong in terms of the pricing data, not the validation algorithm.

**Expensive validation happens once at construction/activation time, not on every calc_price call.** _(from "Validation replaces what hardcoded fields gave us", "Unit definitions travel with prices", "Validation is split between the registry and `set_custom_snapshot`")_
The repo contains thousands of model prices. Loading pre-built data must not become significantly more expensive, must not require price-level validation of every model, and must not require downloading significantly more data. Registry-level structural validation (usage key uniqueness, price key uniqueness, dimension-set uniqueness, interval closure, join-closedness) runs when the `UnitRegistry` is constructed — once per snapshot lifecycle. Price-level validation (ancestor coverage, join coverage, price key validity) is O(k²) over the priced units for each model, where k is that model's priced unit count; it should use registry indexes and relationship caches instead of repeatedly scanning every unit in the registry. Computing decomposition plans overlaps with that same per-model relationship work but is not the same thing as validation and is not a required precomputation: a pre-built model price can be known valid because the repo build already validated it, even though no runtime decomposition plan has been computed.

For repo-defined prices, validation runs in the build pipeline. The build pipeline may exercise the same planning logic to prove correctness, but generated outputs do not include decomposition plans or bulky per-price validation artifacts. The outputs — `data.json`/`data_slim.json` for runtime updates, and generated code like `data.py` (Python) that embeds the same data as language-native structures — contain only units and prices. At runtime, the bundled snapshot (loaded from generated `data.py`) is imported directly — it never passes through full-snapshot price validation. Built-in `ModelPrice` objects may compute/cache decomposition lazily when first priced if that remains straightforward, but correctness must not depend on having precomputed those plans. Official auto-update payloads are likewise treated as repo-validated data after parsing and structural registry checks; activating them must not revalidate every model price. User-created snapshots validate only model prices whose known-valid state is missing or stale, so custom errors are caught early when possible without punishing unchanged built-in prices. If a custom price somehow bypasses activation, `calc_price` still validates that one model before pricing it.

**Unit definitions are generated into language-native code alongside prices.** _(from "Unit definitions travel with prices", "Expensive validation happens once at activation time")_
Each language package has a generated file that embeds all price data as language-native structures — `data.py` for Python, the equivalent for JavaScript. This is what loads at startup, not `data.json`. The `data.json` file exists for runtime auto-updates (refreshing prices in a long-running process). All of these must contain unit definitions. The build pipeline generates unit definitions into the language-native files as code, and into `data.json` as data. When `data.json` is fetched at runtime to update prices, the unit definitions it carries are also loaded. Official fetched snapshots are parsed, structurally checked, and treated as repo-validated for price-level rules without revalidating every model price.

**Price key validation: every key in a model's prices must be a registered price key.** _(from "Validation replaces what hardcoded fields gave us")_
If a provider YAML file has `prices: { inptu_mtok: 3 }`, the build fails. This replaces the implicit validation that hardcoded Pydantic/dataclass fields provided. Checked at build time (data pipeline) and at `set_custom_snapshot` time for runtime-defined prices.

**Dimension names and values are unit-local.** _(from "Dimensions define unit specificity")_
The family does not separately declare allowed dimension keys or values. A dimension typo can therefore create a different axis or value rather than failing a closed-world dimension validation check. That is an acceptable tradeoff in this PR: it removes a separate declaration surface and avoids needing APIs for adding dimension axes or values. Structural validation still checks whether the resulting unit graph is internally usable for pricing (dimension-set uniqueness, interval closure, and join-closedness), and price-level validation still checks key validity, ancestor coverage, and join coverage. A stricter optional dimension schema can be added later if the extra typo protection becomes worth the complexity.

**Usage keys are globally unique across all unit families.** _(from "Validation replaces what hardcoded fields gave us", "Usage is a registry-aware class")_
A usage key identifies a single unit in the entire registry, not just within its family. Usage keys are the raw registry keys, `Usage` attribute names, and extractor destinations — if two unit families could share a usage key, usage extraction and decomposition would be ambiguous.

**Unit uniqueness: no two units in a family may have identical dimension sets.** _(from "Validation replaces what hardcoded fields gave us", "Dimensions define unit specificity")_
Dimensions uniquely identify a unit's position in the containment poset. Two units with the same dimensions would be the same slot — that's a data error, not an edge case.

**Price keys are globally unique across all unit families.** _(from "Validation replaces what hardcoded fields gave us", "ModelPrice supports attribute access")_
A price key maps one-to-one to a unit globally, not just within a family. ModelPrice uses price keys as attribute names — if two units shared a price key, price storage and attribute access would be ambiguous.

**Ancestor coverage is validated.** _(from "Ancestor coverage: pricing a unit requires pricing all its ancestors", "Expensive validation happens once at activation time")_
Checked in the build pipeline for repo-defined prices, and at `set_custom_snapshot` time for runtime-defined prices. Not deferred to calc_price.

**Join coverage is validated.** _(from "Join coverage: pricing two overlapping units requires pricing their intersection", "Expensive validation happens once at activation time")_
For every pair of priced compatible units within the same family, their dimension-union must also be priced. Registry join-closedness guarantees that unit exists. Checked in the build pipeline for repo-defined prices, and at `set_custom_snapshot` time for runtime-defined prices.

**No price sanity inequalities in this change.** _(from "Validation replaces what hardcoded fields gave us", "Existing provider YAML files are mostly unchanged")_
Validation in this PR is limited to correctness rules required by the registry and decomposition algorithm: key validity, dimension-set uniqueness, registry interval closure, registry join-closedness, ancestor coverage, and join coverage. Do not add economic sanity checks such as cache-read ≤ uncached ≤ cache-write or modality price ordering. Those checks may be useful later, but they are scope creep for the unit-registry change.

**Validation rules are expressed in terms of dimensions, not unit names.** _(from "Derive, don't duplicate", "Dimensions define unit specificity")_
All validation logic — interval closure, join-closedness, ancestor coverage, join coverage, dimension-set uniqueness, key matching — operates on the dimension structure and indexes from the registry. No validation code references `input_tokens`, `input_mtok`, or any other specific unit or price key by name.
